## **Parvai Insight: Hardware-AI Internship Task** 

This is my submission for the Parvai Technologies Hardware-AI Bridge Task. It covers YOLO26n inference benchmarking, IMU-based fall detection with a Verilog FSM, an integrated three-thread pipeline, and a hardware architecture proposal, all built and measured on a real Oracle Cloud ARM64 instance.

## **Running each task:** 

## **Task 1 YOLO benchmarks:** 

python3 measure_yolo.py 640 1 python3 measure_yolo.py 416 1 python3 measure_yolo.py 320 1 

Each run takes 60 seconds and saves raw JSON to results/. Consolidated results and power calculations are in task1_fps.md and task1_power_analysis.md. 

## **Task 2 Fall detection:** 

python3 firmware/fall_detector.py 

Generates imu.csv, runs the detector, prints timing stats, and writes results/task2_output.csv and results/task2_fall_plot.png. The underlying data is deterministic (seed=42) so you’ll get the same fall events every time, only the timing numbers vary between runs. 

## **Task 3 Integrated pipeline:** 

python3 tests/test_tts.py _# make sure TTS works before the full run_ nohup python3 ai-pipeline/integrated_pipeline.py 2 > task3_run2_output.log 2>&1 **&** 

Runs for 10 minutes and logs CPU/memory/alerts every 30 seconds. I use nohup because I lost one run when an SSH session dropped partway through backgrounding it means a disconnect won’t kill the process. 

## **Verilog FSM:** 

iverilog -o fsm_sim firmware/fall_detector_fsm.v vvp fsm_sim 

Should print exactly 3 alert lines, one per injected fall in the testbench stimulus. 

## **Things that tripped me up:** 

A few issues I hit during development that aren’t obvious from the code: 

- **espeak crashes on the first TTS call** because the driver’s default voice name doesn’t match what’s actually installed on this Ubuntu image. I fixed it by patching the driver directly details in task3_run_log.md. 

- **aplay: not found** just means alsa-utils isn’t installed. The apt command above covers it. 

- **“terminate called without an active exception”** prints on every exit. It’s harmless results are already written by then. Likely an espeak C-binding cleanup issue, never fully tracked down but confirmed not affecting output across multiple runs. 

- **The Verilog testbench needs one extra sample after fall 3.** The FSM only moves state when sample_valid pulses. Falls 1 and 2 get that pulse for free from the walking stimulus that follows them, but fall 3 is the last event before $finish. I added one trailing sample explicitly to let the FSM complete its final transition it’s a testbench detail, the FSM module itself is fine. 

- **Fall detector timing is unstable between runs** (0.44–0.68µs average, 37–267µs worst case across 5 runs on the same instance with no other load). I checked background CPU and hypervisor steal time, both were zero. Most likely Python scheduler jitter. Reported as a range in task2_timing_analysis.md rather than picking one number. 

## **A note on methodology:** 

For Task 1, I ran each resolution twice to check if the numbers were stable, they were within 1% on FPS and essentially identical on CPU%. The values in the results files come from the cleaner of the two runs per resolution (explained in task1_fps.md). Task 3 was also run twice for the same reason. Run 1’s raw terminal output wasn’t saved after the run finished, so task3_run_log.md only shows Run 2’s complete log, with a note explaining why. 

All numbers in the results files are from real runs on the real Oracle instance. Every calculation shows the formula first, then the actual numbers substituted in, so the arithmetic is checkable rather than just asserted.
